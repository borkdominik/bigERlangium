# bigER Architecture (VS Code + Langium + Sprotty)

## 1) Monorepo structure

```text
bigERlangium/
├─ package.json                      # workspace + top-level build orchestration
├─ packages/
│  ├─ common/
│  │  └─ src/constants.ts            # shared enums/constants (notation + relation types)
│  ├─ language-server/
│  │  ├─ src/entity-relationship.langium
│  │  ├─ src/generated/*             # generated by langium-cli
│  │  ├─ src/entity-relationship-module.ts
│  │  ├─ src/entity-relationship-validator.ts
│  │  ├─ src/diagram/diagram-generator.ts
│  │  └─ src/main.ts                 # LSP server + langium-sprotty handlers
│  ├─ webview/
│  │  ├─ src/main.ts
│  │  ├─ src/di.config.ts
│  │  ├─ src/model.ts
│  │  └─ src/views.tsx               # diagram rendering logic (notation-specific)
│  └─ extension/
│     ├─ src/main.ts                 # VS Code extension activation + language client
│     ├─ src/webview-panel-manager.ts
│     └─ esbuild.mjs                 # bundles extension+server+webview into pack/*
└─ examples/*.er
```

## 3) End-to-end flow (open diagram + live updates)

```text
User
  │
  │ 1) Opens .er file / runs "Open in Diagram"
  ▼
VS Code Extension
  │
  │ 2) LanguageClient starts LSP server (IPC)
  │ 3) ERWebviewPanelManager opens webview + endpoint
  ▼
Webview (Sprotty)
  │
  │ 4) Requests diagram for document URI
  ▼
Language Server (Langium + langium-sprotty)
  │
  │ 5) Parse text using grammar:
  │    entity-relationship.langium -> generated/grammar.ts + generated/ast.ts
  │ 6) Run validators -> diagnostics/markers
  │ 7) ERDiagramGenerator maps AST -> SModel (nodes, edges, labels)
  │ 8) ELK computes layout (server-side layout enabled)
  ▼
Webview (Sprotty)
  │
  │ 9) Renders diagram with custom views (notation/cardinality visuals)
  │
  │ 10) User edits label or triggers diagram edit action
  ▼
Extension endpoint (LspWebviewEndpoint + edit handlers)
  │
  │ 11) Sends workspace/text edits via LSP
  ▼
Language Server + Language Client
  │
  │ 12) Document changes, re-parse, re-validate, regenerate diagram
  └──────────────────────────────────────────────────────► back to step 9 (live refresh)
```

## 4) Diagram generation internals (server vs webview split)

```text
Server-side (semantic -> model):
  AST Model (entities, relationships, notation, attributes)
      -> ERDiagramGenerator
      -> Sprotty model:
         - node:entity, node:relationship
         - edge / edge:partial / edge:inheritance
         - labels (name, cardinality, roles, attributes)
      -> edge metadata for notation rendering:
         notation, connectivity, relationshipType

Webview-side (model -> pixels):
  Sprotty DI config binds model types to views
      -> EntityNodeView / RelationshipNodeView / NotationEdgeView
      -> notation-specific edge decorations:
         Bachman, Crowsfoot, UML aggregation/composition
      -> CSS + SVG rendering
```

## 5) Build and packaging path

```text
Root `yarn prepare`
  -> build:common
  -> build:language-server (includes `langium generate`)
  -> build:webview
  -> build:extension

Extension build (`packages/extension/esbuild.mjs`) outputs:
  pack/extension/src/main.cjs          # extension entry
  pack/language-server/src/main.cjs    # bundled server entry used by LanguageClient
  pack/diagram/main.js + main.css      # webview bundle
  pack/diagram/libavoid.wasm           # copied for router
```
